# fairing — Product Requirements Document

**版本**: 0.2.0-draft
**状态**: 讨论中
**背景**: 作者以 DBOps/AIOps 研发视角，持续研究 AI 时代的职业与技术演变。
订阅源持续增多，需要一套个人化过滤与排序机制，在不依赖云端 LLM 的前提下，
每天从大量资讯中提炼出真正有价值的内容。

---

## 一、问题定义

### 1.1 当前状态
- 21 个订阅源，每天产出约 30～80 篇文章
- 全量进入 Obsidian 和 NotebookLM，无过滤
- 无法区分"改变判断"的内容和"噪音"

### 1.2 核心矛盾
- 关键词过滤太粗暴：无法理解语义，产生大量误杀和漏网
- 同一来源产出的文章质量差异巨大，来源本身不是有效信号
- 用户自己在探索期，无法预先定义准确的偏好规则

### 1.3 解决思路
**用行为代替规则**：通过用户每日对采样文章的 +/- 标注，训练个人分类模型，
让系统从阅读行为中学习用户偏好，而非让用户预先定义偏好。

---

## 二、用户需求

### N1. 内容采集（现有）
- 从配置的 RSS 订阅源每日抓取文章
- URL 规范化 + 标题语义去重，防止重复文章进入流水线
- 可选：通过 Firecrawl 抓取博客全文（用于提升评分质量）

### N2. 全量归档（扩展）
- **所有经过去重的文章** 无论评分高低，均写入 NotebookLM 归档
- NotebookLM 是原始知识库，不应被评分体系污染
- 按周归档，保持结构一致

### N3. 个人化评分
- 对每篇文章计算相关性分数（0～1）
- 分数由个人模型驱动，模型从用户标注数据中学习
- 支持可替换的评分引擎（规则 → 本地模型 → 可选 LLM）

### N4. 过滤与排序输出
- Obsidian 日报 和 邮件：只展示高分文章（Top N），附低优先级标题列表
- 过滤是 Obsidian/邮件层的行为，不影响 NotebookLM 归档
- Top N 数量可配置，默认 15 篇

### N5. 采样反馈机制
- 每日运行后，不自动展示所有文章供评分
- 从当日文章中**采样** 5～8 篇，确保来源和话题多样
- 采样策略强制用户接触不同类型内容，避免确认偏见
- 用户对采样文章打 `+`（有价值）或 `-`（不感兴趣），可跳过
- 标注存入本地 `feedback.jsonl`，不提交 Git

### N6. 偏好衰减
- 标注不删除，通过权重衰减模拟"遗忘"
- 衰减单位：**以已标注文章数量为计数器**，而非日历时间
  - 每新增 3 篇标注，旧标注权重衰减一档
  - 表达的语义：你最近读了 3 篇新的内容，之前的偏好开始淡化
- 模型训练时使用衰减后的权重，旧数据仍然保留在文件中

### N7. 模型训练
- 积累约 30 条标注后可首次训练
- `\train` 命令触发训练，不自动触发
- `\stats` 命令显示标注健康度和训练建议

### N8. 模型架构（可替换）
- **默认**：基于规则的打分（无需训练数据，立即可用）
- **阶段二**：sentence-transformers 向量化 + LogisticRegression（本地，免费）
- **插件预留**：Ollama（本地 LLM）/ Gemini / Claude（云端，可选）
- 通过 `.env` 中 `SCORER=rule|model|ollama|gemini` 切换

---

## 三、约束

| 约束 | 说明 |
|------|------|
| 成本 | 默认零 API 成本，LLM 为可选插件 |
| 数据安全 | 所有个人数据（标注、模型、状态）gitignored |
| 操作简洁 | 反馈机制不能比"每天运行一次脚本"麻烦太多 |
| 渐进演进 | 每个阶段独立可用，无需等待全部完成 |

---

## 四、不在范围内

- 自动化反馈（无点击追踪、无阅读时长统计）
- 多用户 / 团队场景
- 实时推送 / webhook
- 对全文内容的语义理解（仅用 title + excerpt + 可选 snippet）
